NB: ./scripts/macos-pty-investigate.sh has a sed bug that prevents the
test from ever running on macOS. Its sed pattern matches the harness's
own `#[cfg(target_os = "linux")]` on `forkpty_compat`
(crates/rush-cli/tests/pty/mod.rs:386), turning it into `#[cfg(unix)]`,
which then collides with the macOS variant at line 399 ⇒ E0428
duplicate-definition. Build aborts before any test runs. The script
needs a more specific pattern, e.g.
`s/^#!\[cfg(target_os = "linux")\]$/#![cfg(unix)]/` (inner attribute
only), or it should restrict its FILES list to the two test files
(pty_smoke.rs, pty_paint_no_absolute.rs) and skip pty/mod.rs.

Workaround for this run: hand-flip the gates in the two test files
only (pty_smoke.rs:12, pty_paint_no_absolute.rs:19), build, run.

So the harness reached pty_smoke.rs:46 — that's the
`s.expect_exit_within(Duration::from_secs(2))` call after
`s.send_signal(libc::SIGHUP)`. The 2s expect_exit_within elapsed

SIGHUP-driven teardown path of rush-cli plus the harness's reliance
on a blocking waitpid after SIGKILL.

This isolates the bug. SIGKILL → kernel terminates rush-cli before
any userspace teardown runs → waitpid returns immediately → harness
exits cleanly. SIGHUP → rush's signal handler fires → userspace
teardown begins → something wedges → process enters state E → SIGKILL
sent later is silently ignored (kernel doesn't deliver new signals to
processes already in exit teardown) → waitpid blocks forever.

So:
  - The macOS forkpty pty harness is correct.
  - rush-cli starts, paints a prompt, runs an echo, repaints — all
    fine on a forkpty-attached pty on macOS.
  - The bug is in rush-cli's SIGHUP teardown path on macOS only
    (SIGHUP works correctly on Linux: pty_smoke passes there in <0.5s).
  - As a secondary issue, the harness's expect_exit_within Drop /
    SIGKILL path uses a *blocking* waitpid; if the SIGKILL'd child is
    already in kernel state E, that waitpid never returns. Should be
    bounded with WNOHANG + sleep loop, or the harness should accept
    "process is unreachable" as a terminal state.
